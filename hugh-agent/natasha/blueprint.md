Technical Blueprint: Meta-Harness Framework for Autonomous System Optimization  1. The Strategic Paradigm of Harness Engineering  The performance of frontier Large Language Models (LLMs) is no longer a function of weights alone; it is a function of the environment in which those weights are deployed. We are witnessing the industrialization of the "Outer Loop"the code responsible for storage, retrieval, and context presentationnow formally termed Harness Engineering. Research indicates that varying the harness around a fixed model can induce a 6x performance gap. More critically, the 2026 ARC-AGI-3 benchmarks reveal a stark reality: while humans solve these reasoning environments at 100%, frontier models like GPT-5.4 (High) and Gemini 3.1 Pro score nearly 0% (0.26% and 0.37% respectively) without a specialized harness. This frames the harness not as an optimization layer, but as a non-negotiable prerequisite for agentic intelligence.  Harness Engineering represents a shift from manual, heuristic-based prompt adjustment to automated, agentic search. By treating the environment as an executable program, we move beyond simple text-shuffling into a discipline of system architecture where the "diagnostic footprint" of the model informs its own environmental evolution. This transition mandates a modular framework capable of navigating high-dimensional code-space to discover strategies that exceed human design limits.  2. High-Level System Architecture: The Three-Component Loop  Achieving end-to-end harness search requires a modular, outer-loop architecture to manage the complexity of continuous generation and validation. The Meta-Harness framework operates as a perpetual three-component loop:  1. The Proposer (Intelligence Layer): An agentic coding system (specifically Claude Code powered by Opus 4.6) with unrestricted filesystem access. It functions as a digital coworker that navigates, diagnoses, and edits the codebase. 2. The Evaluator (Validation Layer): A frozen base model (M) and a task-specific reward function (r). This layer quantifies the efficacy of candidate harnesses without altering the underlying model parameters. 3. The Filesystem (Memory Layer): A structured directory (D) serving as the system's "long-term memory," housing all prior source code, scalar rewards, and raw execution traces.  The framework is governed by a formal optimization goal to find the optimal harness (H^*) across a task distribution:  H^* = \arg \max_H \mathbb{E}_{x \sim X, \tau \sim p_M(H,x)} r(\tau, x)  This architecture transforms individual failures into a searchable database of experience, allowing the proposer to scale its reasoning beyond the context limits of any single model.  3. The Intelligence Core: Agentic Proposer Design  The strategic necessity of an agentic proposer, rather than a raw LLM, stems from the scale of the diagnostic data. In demanding settings, the proposer must ingest and analyze vast volumes of execution logs that far exceed standard context windows. By using a coding agent with filesystem access, the system can selectively query and process data, mimicking a human engineer's workflow.  Methodology: Selective Diagnostics over Compressed Feedback  Traditional optimization methods (e.g., OPRO or TextGrad) rely on "compressed feedback"scalar scores or short summaries that strip away the causal nuances of failure. Meta-Harness grants the proposer unrestricted access to the filesystem. In practice, the median proposer reads 82 files per iteration, referencing over 20 prior candidates to identify specific failure modes. The Memory Core: Filesystem-Based Experience Logs  In Meta-Harness, the "raw diagnostic footprint" is prioritized over summarized feedback. In complex systems, a failure in reasoning might be caused by a retrieval choice made dozens of steps earlier. Summaries lose this granularity. We mandate a structured directory D where every candidate directory contains:  * source_code.py: The complete, executable harness. * eval_scores.json: Scalar rewards and Pareto metrics (e.g., accuracy vs. context cost). * execution_traces.txt: Raw logs including prompts, tool calls, and exact model outputs.  The Diagnostic Advantage: By utilizing grep across this filesystem, the proposer identifies "confounded edits"multiple simultaneous changesand isolates "causal changes" that drive performance. This auditability is critical because scalar scores cannot detect "chaotic neutral" behaviors, such as a model leaking prompt details into a UI, which can only be identified through raw trace inspection.  5. Execution Logic: The Step-by-Step Search Loop  Algorithm 1 represents a minimalist but powerful evolution in code-space, leveraging the logical structure of programs to regularize the search.  1. Initialization: Seed the population with baseline harnesses (Zero-shot, Few-shot, ACE). 2. Discovery: The proposer queries the filesystem to diagnose regressions across prior candidates. 3. Proposal: The agent generates k new candidate Python files (typically two per iteration). 4. Interface Validation: Automatic check-steps ensure the code is executable and compatible with the system interface. 5. Evaluation: Candidates are run against task distributions (e.g., IMO-level math or text classification). 6. Logging: Full results and traces are written back to a new directory in the filesystem.  A central architectural mandate is the Pareto Frontier. This is not merely a balance but an automated trade-off discovery process. Our OOD data shows that "more context" is often a failing strategy; adding few-shot examples beyond 32 actually hurt performance in 7/9 tasks. The search loop ensures we discover the Pareto-optimized context rather than naively saturating the model window.  6. Performance Benchmarking across Specialized Domains  The framework demonstrates significant cross-domain versatility, yielding gains that transfer to entirely unseen models and datasets.  * Online Text Classification: Meta-Harness delivered a 7.7-point improvement over ACE while utilizing 4x fewer context tokens. It reached the peak performance of prior optimizers in just 10% of the evaluation time. * Retrieval-Augmented Math Reasoning: On 200 IMO-level problems, the framework achieved a 4.7-point average gain. Notably, this was achieved using a BM25-based lexical retrieval stack rather than a dense encoder, proving that architectural nuance in the harness outweighs raw embedding complexity. These gains were verified across held-out models including Gemini 3.1 and GPT-5.4. * Agentic Coding (TerminalBench-2): Discovered harnesses ranked #1 for Haiku 4.5 and #2 for Opus 4.6, surpassing established hand-engineered agents like Terminus-KIRA.  These results are not the product of overfitting. The discovered harnesses capture generally effective strategiessuch as refined state managementthat maintain their efficacy even when the underlying base model is swapped.  7. Deployment Protocols: Security and Coding Best Practices  Autonomous code generation requires strict guardrails to maintain project vision and safety. Meta-Harness implements the following non-negotiable protocols:  * Restricted Action Space: The agent is forbidden from modifying the base model (M) or core evaluation logic; it is confined strictly to the harness. * Interface Validation: Mandatory checks prevent broken code from polluting the experience logs. * Regularization Bias: Code-space search naturally prevents the "brute-force" if-chaining seen in weight-based optimization, as coding agents favor reusable algorithmic structures. * Auditability and Safety: Leveraging raw filesystem traces is essential for identifying "chaotic neutral" behaviors. Reports on GPT-5.4 indicate tendencies to implement unrequested features, such as adding GDPR consent checkboxes to simple demos or leaking prompt details into UI elements. A scalar reward would overlook these rogue behaviors; only a filesystem-based "grep-able" audit trail allows for human-in-the-loop inspection of such outputs.  The Meta-Harness value proposition is clear: by transforming the "diagnostic footprint" into a structured, agentic search, we can automate the engineering of the Outer Loop, delivering superior, transferable, and auditable system performance.
