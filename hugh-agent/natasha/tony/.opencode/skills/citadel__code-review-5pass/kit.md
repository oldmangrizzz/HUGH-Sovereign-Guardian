Formatters catch formatting. Type checkers catch type errors.\nNone of them catch the things that actually break production.\n\nThis kit fills that gap. It runs five structured passes over your code to find\nproblems that require genuine comprehension to detect: a race condition between\ntwo async operations, an unparameterized query that looks like string formatting,\nan O(n²) loop that only matters when your dataset grows, a comment that hasn't\nmatched the code since the refactor six months ago.\n\nThe output is a structured report with every finding numbered, located to the exact\nline, and accompanied by a specific fix — not \"consider improving this\" but \"change\nline 47 from X to Y because Z.\"\n\nA verdict at the end (PASS / CONDITIONAL / FAIL) gives you the bottom line in one\nword, with finding counts if you need to justify a review decision.\n\n## When to Use\n\n- Before merging any non-trivial pull request\n- When taking over code written by someone else (or your past self)\n- After a fast-moving sprint where correctness was deprioritized for speed\n- Before a security audit — catch the obvious issues yourself first\n- When a system is behaving unexpectedly and you suspect the bug is subtle\n\n**Skip this kit for:** generated files, lock files, configuration with no logic,\nand code so simple the 5-pass structure would produce empty passes.\n\n**Use `/review --diff main..feature`** for PR reviews — it focuses on changed lines\nand their context, distinguishing new-code findings from pre-existing issues.\n\n## Setup\n\nNo setup required. The kit reads from your codebase directly.\n\n**Recommended:** have your project's config files at the repo root before running:\n- TypeScript: `tsconfig.json`\n- ESLint: `.eslintrc.*` or `eslint.config.*`\n- Prettier: `.prettierrc.*`\n- Python: `pyproject.toml`\n- Rust: `Cargo.toml`\n\nThese are loaded in Step 2 to establish what \"correct\" and \"consistent\" means for\nyour project specifically. Without them, the review still runs — it just uses\ngeneral best practices instead of your project's conventions.\n\n## Steps\n\n### Step 1: Resolve Scope\n\nDetermine what's being reviewed based on the input:\n\n- **File path**: read that file\n- **Directory**: glob for all source files recursively, skip generated files,\n  `node_modules`, lock files, and build artifacts\n- **Diff range**: run `git diff <range>` to get changed files and hunks, then read\n  the full file for each changed file (context beyond the hunks matters)\n- **No argument**: run `git diff HEAD` for staged + unstaged changes\n\nRead all files in scope before starting any pass. Do not re-read during individual passes.\n\n---\n\n### Step 2: Load Project Conventions\n\nRead the project's config files to understand its standards before reviewing against them:\n- What import style does it use? (Path aliases? Relative paths? Named vs. default exports?)\n- What error handling pattern does it use? (throw vs. Result types vs. callbacks)\n- What naming conventions does it follow?\n- What test patterns exist?\n\nThese become the baseline for Pass 5 (Consistency). Flag deviations from them, not from\ngeneric \"best practices\" that may not apply to this project.\n\n---\n\n### Pass 1: Correctness\n\nScan every file for:\n\n- **Logic errors**: inverted conditions, wrong operators, incorrect boolean logic\n- **Off-by-one errors**: in loops, slices, index access, range calculations\n- **Null/undefined dereference**: accessing properties without guards when the value\n  could be null or undefined\n- **Missing awaits**: async calls whose results are used without awaiting them\n- **Race conditions**: shared mutable state accessed from concurrent async operations\n  without synchronization\n- **Type coercion**: loose equality (`==`), implicit type conversions, surprising\n  JavaScript truthiness/falsiness\n- **Resource leaks**: connections, handles, event listeners, or subscriptions opened\n  but never closed\n- **Missing cleanup**: effects or lifecycle methods that set up state but don't\n  tear it down\n- **Edge cases**: what happens with an empty array? A zero value? A negative number?\n  An extremely large input?\n- **State mutations**: direct mutation of state that's supposed to be immutable\n\n---\n\n### Pass 2: Security\n\nScan for OWASP Top 10 and common vulnerabilities:\n\n- **Injection**: SQL, NoSQL, command, template, LDAP — any user input reaching a\n  query or command without parameterization\n- **XSS**: user input rendered as HTML without sanitization, `dangerouslySetInnerHTML`,\n  `innerHTML`, unescaped template interpolation\n- **Broken auth**: missing authentication checks on endpoints, broken access control,\n  privilege escalation paths, JWT validation gaps\n- **Hardcoded secrets**: API keys, tokens, passwords, or connection strings in source\n  code instead of environment variables\n- **Unsafe deserialization**: `eval()`, `Function()`, `JSON.parse` on untrusted input\n  without schema validation, `pickle.loads`, `yaml.load` without SafeLoader\n- **SSRF**: user-controlled URLs passed to fetch/request without allowlist validation\n- **Path traversal**: user input in file paths without sanitization\n- **Insecure crypto**: MD5/SHA1 for passwords, ECB mode, hardcoded IVs, `Math.random()`\n  for security-sensitive values\n\n---\n\n### Pass 3: Performance\n\nScan for measurable performance problems — not micro-optimizations, but issues that\ncause degradation at scale:\n\n- **Algorithmic**: O(n²) or worse in paths that scale with data size\n- **Allocation in hot paths**: creating objects or arrays inside render functions,\n  animation loops, or frequently-called functions when they could be hoisted\n- **Missing memoization**: expensive derivations recomputed on every call when inputs\n  haven't changed\n- **N+1 queries**: database or API calls inside loops instead of batched operations\n- **Unnecessary bundle weight**: importing entire libraries for one function\n- **Render performance** (frontend): new object/array references causing unnecessary\n  re-renders, missing memoization on expensive derived values, inline function props\n  recreated on every render in high-frequency components\n- **I/O in hot paths**: synchronous reads, blocking operations, or layout-forcing DOM\n  calls inside animation loops\n- **Unbounded queries**: queries or list renders with no pagination or limits\n\n---\n\n### Pass 4: Readability\n\nScan for code that will cost the next developer (or you, in three months) extra time\nto understand:\n\n- **Vague naming**: `data`, `info`, `result`, `handle`, `process` — names that describe\n  shape but not meaning\n- **Misleading naming**: `isValid` returning a string, `getUser` having side effects\n- **Function length**: functions over 50 lines doing multiple distinct things\n- **Cognitive complexity**: three or more levels of nesting, complex boolean expressions\n  that could be extracted to named variables\n- **Dead code**: unreachable branches, commented-out blocks, unused variables and imports\n- **Stale comments**: comments that describe what the code did before a refactor\n- **Magic values**: hardcoded numbers or strings without named constants\n- **Mixed abstraction levels**: high-level orchestration and low-level implementation\n  interleaved in the same function\n\n---\n\n### Pass 5: Consistency\n\nScan against the project conventions loaded in Step 2:\n\n- Does the code follow the project's import ordering and alias usage?\n- Does it use the project's error handling pattern?\n- Does it follow the project's file organization conventions?\n- Do new functions follow the existing signature and return type patterns?\n- Do new identifiers follow the project's naming conventions?\n\nAlso flag *internal inconsistency* — if some functions in a module throw and others\nreturn null for errors, that's a finding even without an established convention.\n\n---\n\n### Format Every Finding\n\nEach finding must include:\n- **File**: the path\n- **Line**: exact line number or range\n- **Severity**: `CRITICAL`, `WARNING`, or `INFO`\n- **Finding**: one sentence describing the problem\n- **Code**: the specific problematic lines\n- **Fix**: what to do about it — specific, not vagues\n\n**Severity calibration:**\n- `CRITICAL`: will cause bugs, security vulnerabilities, data loss, or crashes\n- `WARNING`: causes problems under specific conditions, degrades performance,\n  or creates meaningful maintenance burden\n- `INFO`: style improvement, minor clarity gain, preventive suggestion\n\n---\n\n### Produce the Verdict\n\n```\n## Code Review: {target}\n\nScope: {N files, M total lines} | Mode: {file | directory | diff}\n\n---\n\n### Pass 1: Correctness\n{findings or \"No findings.\"}\n\n### Pass 2: Security\n{findings or \"No findings.\"}\n\n### Pass 3: Performance\n{findings or \"No findings.\"}\n\n### Pass 4: Readability\n{findings or \"No findings.\"}\n\n### Pass 5: Consistency\n{findings or \"No findings.\"}\n\n---\n\n## Verdict: {PASS | CONDITIONAL | FAIL}\n{one-line rationale}\n\n| Severity | Count |\n|---|---|\n| Critical | N |\n| Warning  | N |\n| Info     | N |\n```\n\nFor diff reviews: distinguish findings in new/changed code from pre-existing issues\nsurfaced by context. Prioritize the new-code findings.\n\n---\n\n## Constraints\n\n- **Don't duplicate linter output.** Don't flag things the project's linter or formatter\n  catches automatically (missing semicolons, indentation, unused vars that ESLint flags).\n  Focus on semantic issues that require comprehension to detect.\n- **No false positives from skimming.** For every finding, verify you read the surrounding\n  code. A \"missing null check\" that's actually guarded by the caller is not a finding.\n  An \"unused import\" that's used in a type annotation is not a finding.\n- **The review is the deliverable.** Don't offer to fix findings unless asked. The review\n  surfaces issues; the developer decides which ones to address and how.\n- **Line numbers must be accurate.** A finding pointing to the wrong line is worse than\n  no finding. Verify each cited line against the actual file content.\n\n## Safety Notes\n\n- Pass 2 (Security) findings rated CRITICAL should be treated as blocking for production\n  deployments. Don't ship CRITICAL security findings.\n- When reviewing authentication or authorization logic, read the entire auth flow, not\n  just the changed file — a bypass may be in an upstream guard, not the file being reviewed.\n- If the review surfaces hardcoded credentials, do not include the actual credential value\n  in the report — just the location and the finding.\n
