The first time\nwith incomplete information, and the second time after the consequences arrive.\n\nThis kit doesn't produce comprehensive literature reviews. It produces *decision-ready*\nfindings: the specific facts that matter for the question at hand, confidence-labeled\nso you know how much to trust them, with source citations so you can verify them,\nand with explicit action recommendations so the research connects to a decision.\n\nThe key discipline is focus. 2-4 queries, 3-6 credible sources, a written document,\ndone. Not exhaustive. Timely. The research document persists, so if the same question\ncomes up again in three months, the prior findings are available without re-running\nthe investigation.\n\n**Single question mode:** one agent investigates a focused question sequentially —\nformulate queries, search, read actual pages, extract findings, write the document.\n\n**Multi-scout mode:** when a question has 3+ genuinely independent angles (e.g., \"Should\nwe migrate from Express to Fastify?\" breaks into performance benchmarks, migration effort,\ncommunity health, and production war stories), multiple scout agents investigate their\nangles simultaneously. Findings are compressed between waves into a unified brief. Faster\nfor broad questions; overkill for narrow ones.\n\n## When to Use\n\n**Single question mode:**\n- Checking whether a dependency has a newer version or has been superseded\n- Finding best practices for a specific technical problem\n- Reading official documentation for an API or library before using it\n- Verifying whether a pattern in the codebase has known issues\n- Any time you need external information before making a code decision\n\n**Multi-scout mode:**\n- Evaluating multiple competing technologies or approaches\n- Any question that naturally decomposes into 3+ independent sub-questions\n- Research where parallel investigation would produce meaningfully better results\n  than single-thread depth\n\n**Use local-only mode when:**\n- Researching internal patterns (\"how does our codebase handle auth errors?\")\n- Checking existing library usage without needing external sources\n- Air-gapped environments or when web search isn't appropriate\n\n## Setup\n\nNo setup required for web-based research. The kit uses WebSearch and WebFetch\nto find and read sources.\n\nFor local-only research, the kit uses Grep and Read to search the codebase.\nNo additional setup.\n\n`.planning/research/` is created automatically on first run if it doesn't exist.\n\n## Steps\n\n### Step 1: Formulate\n\nConvert the research question into 2-4 specific search queries. For each query,\ntarget a different source type:\n\n```\nOfficial docs query:    \"express.js middleware error handling site:expressjs.com\"\nCommunity/GitHub query: \"express error middleware best practices 2025\"\nTechnical comparison:   \"express vs fastify error handling performance\"\nRelease notes query:    \"express 5.x changelog breaking changes\" (if version-specific)\n```\n\nState the question clearly in one sentence before searching. This prevents scope\ndrift during the investigation.\n\n---\n\n### Step 2: Search and Read\n\nExecute searches and read actual content — not just snippets:\n\n- Use WebSearch for discovery (snippets tell you which pages to read)\n- Use WebFetch to read the actual pages that look relevant\n- Evaluate source credibility: official docs > GitHub repos > recent technical posts > forum answers\n- Stop at 3-6 credible sources. Not exhaustive — focused.\n- Note disagreements between sources rather than silently resolving them\n\n**Do not fetch GitHub repository pages directly** — they're large HTML documents\nthat time out. Instead:\n- Use WebSearch to find information about repos (snippets contain what you need)\n- For README content, fetch the raw URL:\n  `https://raw.githubusercontent.com/{user}/{repo}/{branch}/README.md`\n\n---\n\n### Step 3: Extract Findings\n\nFor each finding:\n\n```\nWhat:       The specific fact, recommendation, or pattern discovered\nSource:     URL or reference\nRelevance:  How this applies to the original question (one sentence)\nConfidence: high (official docs, verified) | medium (community consensus) | low (single source)\nAction:     What to do with this information, or \"informational only\"\n```\n\nDon't record every interesting thing found — only findings that bear on the\noriginal question. Research scope creep produces long documents nobody reads.\n\n---\n\n### Step 4: Write the Research Document\n\nWrite findings to `.planning/research/{topic-slug}.md`:\n\n```markdown\n# Research: {Topic}\n\n> Question: {The original question}\n> Date: {ISO date}\n> Confidence: {overall: high/medium/low}\n\n## Findings\n\n### 1. {Finding title}\n**What:** {description}\n**Source:** {URL}\n**Confidence:** {high/medium/low}\n**Action:** {recommendation or \"informational\"}\n\n### 2. {Finding title}\n...\n\n## Summary\n{2-3 sentences: what was learned, what the recommendation is}\n\n## Open Questions\n{Anything that couldn't be resolved — needs human judgment or deeper investigation}\n```\n\n---\n\n### Step 5 (Multi-Scout Mode Only): Decompose, Deploy, Compress\n\nIf the question has 3+ independent angles:\n\n**Decompose** into specific sub-questions, each:\n- Independent (doesn't depend on another scout's findings)\n- Specific (one clear question)\n- Answerable in 3-6 sources\n\n**Deploy scouts** in parallel, one per angle. Each follows the single-question\nprotocol (Steps 1-4) and writes its findings to\n`.planning/research/fleet-{slug}/{angle-slug}.md`.\n\n**Compress** after all scouts complete:\n- Consensus: findings independently confirmed by multiple scouts\n- Conflicts: findings that contradict (flag prominently, don't resolve silently)\n- Gaps: angles without strong results\n- Surprises: unexpected findings that change the question's framing\n\nIf gaps or conflicts exist, deploy a Wave 2 with targeted scouts to resolve them.\nWave 2 scouts receive the compressed brief as context — they don't re-research\nwhat Wave 1 already covered.\n\n**Write the unified report** to `.planning/research/fleet-{slug}/REPORT.md`.\n\n---\n\n## Constraints\n\n- **Does not make decisions.** The kit produces findings and a recommendation.\n  The decision is yours. Architecture choices, library adoptions, and migration\n  commitments require human judgment — the kit informs, not decides.\n- **Does not install packages or modify code.** Research is read-only. Apply\n  findings separately.\n- **2-4 queries, 3-6 sources, done.** Not exhaustive. If the question requires\n  deeper investigation, run a second research pass on the specific open questions\n  from the first.\n- **Maximum 5 scouts per wave** in multi-scout mode (diminishing returns beyond that).\n- **Maximum 2 waves** — if Wave 2 doesn't resolve conflicts, the question needs\n  human judgment, not more searches.\n\n## Safety Notes\n\n- Research findings are only as good as their sources. `Confidence: high` means\n  the finding comes from official documentation or has multiple corroborating\n  sources — not that it's certainly correct. Verify critical decisions independently.\n- The research document is persistent. If you run research on a topic and then\n  the landscape changes (library deprecation, new version, breaking change), the\n  old document may be stale. Check the date before acting on old research.\n- For security-sensitive research (vulnerability assessment, crypto library selection,\n  auth pattern evaluation), treat `Confidence: medium` findings with more skepticism\n  than you would for other topics. Security mistakes compound.\n
