It improves through iteration — scoring\nwhat exists, picking the worst thing, fixing it, and doing it again. The problem\nis that iteration requires sustained attention. Sessions end. Context resets.\nThe loop stalls.\n\nThis kit automates the iteration. It runs a tight cycle: score everything against\na rubric, select the single highest-leverage axis, attack it with a real fix,\nverify the fix worked and nothing regressed, document what was learned, then loop.\nEach iteration re-scores from scratch — because what you fixed this loop changes\nthe priority order for the next.\n\nThe rubric is the key design choice. It's not a checklist. Each axis has three\nanchors (what a 0 looks like, what a 5 looks like, what a 10 looks like), a weight,\nand programmatic verification specs where possible. This makes scores arguable,\nnot vibes-based. Three independent evaluators score each axis and the minimum\nscore wins — because a low score from any evaluator is an unresolved problem, and\naveraging would hide it.\n\nThe loop stops when you tell it to, when a quality ceiling is reached (all axes >= 8.0),\nor when the rubric has been extracted to its ceiling and needs to be re-anchored at\na higher standard.\n\n## When to Use\n\n- A product, module, or codebase that needs systematic quality improvement over time\n- After shipping a fast MVP that you know has rough edges but aren't sure which to\n  address first\n- When you want to improve a specific dimension (documentation, test coverage,\n  security posture) but aren't sure where the gaps actually are\n- As a continuous improvement process across multiple sessions or overnight runs\n  (pairs with the Daemon kit for unattended operation)\n\n**Not for:** quick bug fixes (use Systematic Debugging), one-off code reviews\n(use the Review kit), or simple refactors. The loop has overhead. It earns its\ncost on targets where the improvement space is wide and the priority isn't obvious.\n\n**Good first run:** use `--score-only` to get your baseline before committing to\nimprovement loops. You'll see where you actually stand, which makes the case for\nhow many loops are warranted.\n\n## Setup\n\n**Create a rubric before running your first loop.**\n\nThe rubric is a markdown file at `.planning/rubrics/{target}.md`. If it doesn't exist,\nthe kit's Phase 0 drafts one and asks for your approval.\n\nA good rubric has:\n- 8-14 axes organized into 3-5 categories (don't go narrow, don't go sprawling)\n- Each axis with a weight (0.0-1.0), a category, and three anchors (0/5/10)\n- Programmatic verification specs for at least half the axes (things that can be\n  checked with a script, not just judged by eye)\n\n**Why approval is required:** the rubric defines what \"better\" means. If the axes are\nwrong, the loop optimizes the wrong things. Phase 0 drafts; you decide.\n\n**Directory structure:** the kit uses `.planning/` for state. Create it at your project root\nif it doesn't exist, or let Phase 0 create it on first run.\n\n## Steps\n\n### Phase 0: Rubric Bootstrap (first run only, requires your approval)\n\nRuns only when `.planning/rubrics/{target}.md` doesn't exist.\n\n1. If research on comparable products exists in `.planning/research/`, reads it\n2. If not, runs a multi-scout research pass to understand the quality landscape\n3. Drafts 8-14 axes with weights, categories, anchors, and verification specs\n4. Presents the draft with rationale for each axis\n5. **Stops. Waits for your approval.** You edit the rubric, confirm it, and the loop begins.\n\nThis is the only mandatory human gate. Everything after this is autonomous.\n\n---\n\n### Phase 1: Score (every loop, no exceptions, no caching)\n\nThree independent evaluator agents score every axis in the rubric simultaneously.\nEach evaluator receives:\n- The full rubric with axis definitions and anchors\n- Read access to the target\n- Their assigned persona (Experienced User, Newcomer, or Critical Reviewer)\n- Instructions to score independently — they don't see each other's scores\n\n**For each axis, the final score is the minimum of the three evaluators.** Not the average.\nA score of 6, 8, 7 produces a final score of 6. A low score from any evaluator\nmeans there's a real unresolved problem somewhere. The minimum surfaces it.\n\nProgrammatic checks run in parallel with evaluator scoring. A failed programmatic\ncheck caps the axis at 5, regardless of evaluator scores.\n\nThe scorecard:\n```\nAxis                    | A  | B  | C  | Prog | Final | Delta\n------------------------|----|----|----|----- |-------|------\nsecurity_posture        | 7  | 8  | 6  | PASS |  6.0  |\ntest_coverage           | 4  | 3  | 5  | FAIL |  3.0  | cap\ndocumentation_accuracy  | 6  | 6  | 7  | PASS |  6.0  |\n```\n\n---\n\n### Phase 2: Select\n\nChoose the single axis to attack this loop using a selection formula:\n\n```\nscore(axis) = (10 - current_score) × weight × effort_multiplier × recency_penalty\n```\n\n- **effort_multiplier**: low-effort changes = 1.0, medium = 0.7, high-effort = 0.4\n  (lower-effort improvements get priority — more ground covered per loop)\n- **recency_penalty**: 0.5 if this axis was attacked in either of the last two loops\n  (prevents fixation on one axis while others deteriorate)\n\nThe selection and its rationale are shown before any attack begins. You can override\nwith `--axis {name}` to force a specific axis regardless of scoring.\n\n---\n\n### Phase 3: Attack\n\nExecute the improvement. Strategy depends on the axis category:\n\n- **Technical axes** (test coverage, reliability, API consistency): run targeted\n  experiments, measure before/after, commit only if improvement is verified\n- **Documentation axes**: read current docs, identify specific gaps or inaccuracies,\n  rewrite the specific sections — not wholesale rewrites unless the score is below 3\n- **Experience axes** (onboarding, error recovery, discoverability): structural fixes\n  plus documentation updates plus verification of the actual user path\n- **Security axes**: read the specific code involved, make targeted changes, run\n  programmatic checks before committing\n\nEach approach is tried in isolation before committing. If multiple approaches are\nevaluated, the decision record is written to the loop log — future loops can read\nwhy the winner won.\n\n---\n\n### Phase 4: Verify\n\nAfter the attack, re-score only the targeted axis (not a full re-score — that's Phase 1\nof the next loop):\n\n1. **Programmatic**: run the axis's specific checks — do they now pass?\n2. **Structural**: verify structural requirements are met\n3. **Perceptual**: spawn a single Newcomer evaluator (the hardest-to-satisfy persona)\n   to score the targeted axis independently\n4. **Behavioral simulation** (for onboarding/UX axes): follow the actual user path\n   in a clean environment with no prior knowledge — does it work?\n\n**Regression check**: re-run programmatic checks on every axis that shares files with\nthe changes. If anything that was passing now fails: **revert the changes**. Don't commit\nimprovements that break something else.\n\nA behavioral FAIL overrides a passing perceptual score. If the user path is broken,\nthe loop doesn't commit regardless of what the evaluator said.\n\n---\n\n### Phase 5: Document\n\nEvery loop gets a log at `.planning/improvement-logs/{target}/loop-{n}.md`.\nEven loops that end in no-change or revert. Especially those.\n\nThe log records:\n- The full scorecard with deltas from the previous loop\n- What was changed and why that approach was chosen\n- Verification results across all four tiers\n- What was learned that future loops should know\n\nThis institutional memory is what makes later loops smarter than earlier ones.\n\n---\n\n### Phase 6: Loop or Exit\n\nThe loop exits when:\n1. A loop count (`--loops N`) is specified and N loops have completed\n2. All axes score >= 8.0 (quality ceiling reached)\n3. No axis improved > 0.5 in either of the last two loops and at least 3 loops\n   have completed (plateau — Level-Up Protocol is triggered)\n4. You say stop\n\n**On plateau (Level-Up Protocol):** the current rubric has been extracted to its ceiling.\nThe kit freezes the current state, proposes re-anchored axes where the current 10 becomes\nthe new 5, and waits for your approval before continuing. This is how the quality bar\ngets raised, not just approached.\n\n---\n\n## Constraints\n\n- **Phase 0 requires your approval.** The rubric is the only thing that requires human\n  judgment. Once it's set, the loop runs autonomously.\n- **The loop never writes to the live rubric.** Proposed axis additions or re-anchorings\n  go to `.planning/rubrics/{target}-proposals.md`. You approve before they take effect.\n- **Level-Up Protocol requires your approval.** The loop halts and waits. It doesn't\n  self-upgrade the rubric.\n- **Regressions block commits.** If the fix breaks something that was passing, the change\n  is reverted. The loop continues without a commit for that iteration.\n- **Security axis failures are blocking.** A failed security programmatic check halts\n  the entire loop — it doesn't get treated as \"just a low score.\"\n\n## Safety Notes\n\n- Use `--score-only` first to see your baseline before spending tokens on attack loops.\n  Knowing your scores in advance helps you calibrate how many loops are warranted.\n- Each loop commits its changes separately. If a loop produces an unwanted result,\n  `git revert <commit>` undoes that loop's changes without touching others.\n- For overnight / unattended runs, pair this kit with the Daemon kit, which handles\n  session continuity and budget enforcement automatically.\n- Set a loop count (`--loops 3`) for your first run rather than running until ceiling.\n  It's easier to add loops than to undo them.\n
